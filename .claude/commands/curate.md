# /curate - 网站合集组装 Agent

## Purpose

从 inbox 网站池中组装主题合集。**前置条件：`/pickdiscover` 已完成网站建档**，`sites/` 目录下有完整档案。核心能力是**读取档案、灵活组织主题**——既能做常规分类（开发工具、设计资源），也能做创意主题（蓝色美学、秋日氛围、探索未知）。

## 运行模式

Agent 模式。触发后自主完成分析和组装，在需要用户决策时暂停询问。全程中文沟通。

## Input

```
/curate                    # 自动分析 inbox，提出主题方案让用户选
/curate 蓝色              # 围绕指定主题组装
/curate AI coding tools    # 围绕指定主题组装
```

## 文件结构

```
picks/
├── inbox.md              # 网站池（快速参考）
├── sites/                # 网站详情档案（一个网站一个文件）
│   ├── bolt-new.md
│   ├── dify.md
│   └── ...
├── drafts/               # 草稿 picks
│   └── {slug}.json
└── picks.json            # 🔒 已发布
```

## Instructions

### Phase 1: 读取网站档案

> 网站建档已由 `/discover` 在收集阶段完成。Curate 只需读取现有档案。

#### Step 1.1: 加载档案

读取 `picks/sites/` 目录下所有 `.md` 档案文件，提取每个网站的氛围关键词、可归类方向等信息。

同时读取 `picks/inbox.md` 获取网站列表和 In Picks 状态。

#### Step 1.2: 检查档案完整性

如果 inbox.md 中有网站在 `sites/` 目录下找不到对应档案，**跳过该网站并提醒用户**（建议先跑 `/pickdiscover` 或 `/collect` 补全）。不要自行抓取和建档。

### Phase 2: 主题提案

#### Step 2.1: 如果用户指定了主题

直接跳到 Phase 3，用指定主题筛选网站。

#### Step 2.2: 如果用户没指定主题

**前置分析**：
1. 读取所有 `picks/sites/` 档案，提取网站的特征维度
2. 读取 `picks.json`，了解已发布主题，**避免重复**
3. 读取 `inbox.md` 的 `In Picks` 列，统计网站使用频率

**主题发散策略**：

主题不应局限于功能分类，从多个维度切入：

- **功能维度**：开发工具、设计资源、数据分析
- **色彩/视觉**：蓝色系网站、暗色美学、极简主义
- **情绪/感受**：安静的工具、让人惊喜的网站、治愈系
- **场景/时刻**：周末探索、深夜阅读、通勤路上、周一提效
- **理念/价值观**：独立开发者精神、隐私至上、开源力量、反 AI
- **用户群体**：给内容创作者的、Solo 开发者的、设计师的
- **对比/反叛**：大厂替代品、被低估的宝藏、反主流工具
- **时间轴**：最近一个月最火的、经典不过时的、复古网络美学
- **问题/痛点**：解决拖延的、保护专注的、简化流程的
- **关系/组合**：工具的工具、平台的平台、生态系统的入口

提出 **5 个主题方案**，要求：
- **多样性优先**：5 个主题应来自不同维度，不要都是功能分类
- **包容度**：既要有稳妥选项（常规主题），也要有实验性选项（创意主题）
- **新鲜度**：检查已发布主题，确保新主题角度不重复
- **可行性**：每个主题标注可选网站数量（至少 8 个），其中至少 5 个未被使用或使用 ≤1 次
- **吸引力**：标题要让人想点开，描述要说清楚"为什么有趣"

输出格式：
```
## 主题方案

1. **{主题名}** — {一句话解释为什么这个主题有趣}
   维度：{功能/色彩/场景/理念等}
   可选网站：Site A (未用), Site B (用过1次), Site C (未用)... (N 个，标注使用次数)
   新鲜度：❌ 与已发布主题 X 有重叠 / ✓ 全新角度

2. ...
```

然后 **暂停等用户选择**（使用 AskUserQuestion tool）。

### Phase 3: 组装 Draft

#### Step 3.1: 筛选网站

根据选定主题，从 sites/ 档案中挑选最匹配的 8-10 个网站。

**存活检查（必做）**：对每个候选网站，用 WebFetch 验证网站是否能正常访问。排除以下情况：
- 返回错误（403/404/500/超时）
- 显示维护页面（maintenance、coming soon）
- 内容与档案描述严重不符（产品已转型或关闭）

如果候选网站不可用，从备选中替换，并在 inbox.md 的 Note 中标注（如"已停机"）。

**选择标准（按优先级排序）**：

1. **网站可正常访问**（前提条件，必须通过存活检查）

2. **使用频率平衡**（核心规则）：
   - 读取 inbox.md 的 `In Picks` 列，统计每个候选网站已被使用的次数
   - **优先选择使用次数 ≤ 1 的网站**（未用过或只用过一次）
   - 避免选择使用次数 ≥ 3 的网站（除非主题强相关且无替代）
   - 目标：让更多网站有机会被看到，而不是同一批网站反复出现

3. **发掘未知宝藏**（重要原则）：
   - 在契合度相近的情况下，**优先选择小众/独立/新兴网站**
   - 避免只选知名大站（如 Awwwards、Product Hunt、GitHub 等）
   - 好的组合：2-3 个"锚点网站"（稍微知名）+ 5-7 个"发现惊喜"（小众好物）
   - 判断依据：网站流量估算、品牌知名度、在档案中的描述

4. **与主题的契合度**：
   - 核心匹配 > 边缘匹配
   - 每个网站在主题语境下都要有存在价值

5. **多样性和层次感**：
   - 不要全是同一类工具
   - 有不同的使用场景、不同的受众、不同的风格
   - 总数控制在 8-10 个

**筛选后输出统计**：
```
使用频率分布：
- 首次使用：5 个
- 使用 1 次：3 个
- 使用 2 次：2 个
- 使用 ≥3 次：0 个 ✓
```

#### Step 3.2: 生成 Draft

写入 `picks/drafts/{slug}.md`（Markdown 格式，方便人工阅读和修改，发布时再转 JSON）：

```markdown
---
status: draft
slug: theme-slug-here
date:
coverImage:
---

# English Title

English description, 1-2 sentences.

# 中文标题

中文描述，1-2 句话。

## Sites

### Site Name
- **URL**: https://...
- **EN**: Why this site matters in this collection.
- **ZH**: 为什么这个网站值得在这个合集里。

### Another Site
- **URL**: https://...
- **EN**: ...
- **ZH**: ...
```

**文案要求：**

标题（title）：
- 英文：有吸引力，不超过 60 字符，适合 SEO
- 中文：自然，不是英文直译，适合中文读者
- 避免 AI 味词汇：leverage、empower、seamless、delve、streamline

描述（description）：
- 说人话，像朋友推荐一样
- 具体，不空洞
- 英文和中文各自独立写，不是互译

网站描述（site description）：
- **不是**介绍这个网站是什么（那是 sites/ 档案的事）
- **而是**这个网站在这个主题下为什么值得关注
- 每个网站的描述要结合主题语境，不是通用说明

#### Step 3.3: 更新 inbox.md

更新被选中网站在 inbox.md 中的 `In Picks` 列：
- 如果是 `-`，改为 `{slug}`
- 如果已有其他 pick，追加 `, {slug}`

#### Step 3.4: 输出预览

输出 draft 的人类可读预览：

```
## Draft 预览: {中文标题}

**Slug**: {slug}
**网站数**: N

| # | Site | 为什么选 |
|---|------|---------|
| 1 | ... | ... |

**英文标题**: ...
**英文描述**: ...

**中文标题**: ...
**中文描述**: ...
```

然后提示用户：
- 可以要求调整选站
- 可以要求修改文案
- 确认后用 `/seo-review` 和 `/copy-review` 做质量检查
- 最终用 `/publish-pick` 发布

## 质量标准

### 主题的好坏

好的主题：
- 让人想点开看看有什么网站
- 有明确的受众（谁会对这个感兴趣）
- 网站之间有化学反应，不是随机拼凑

坏的主题：
- 太宽泛（如 "好用的工具"）
- 太窄（凑不到 8 个网站）
- 没有吸引力（如 "各类网站合集"）

### 创意主题的灵感方向

不要局限于功能分类，可以从这些角度切入：
- **色彩**：蓝色系网站、暗色美学
- **感受**：安静的工具、让人惊喜的网站
- **场景**：周末探索、深夜阅读、通勤路上
- **理念**：独立开发者精神、隐私至上、开源力量
- **对比**：大厂替代品、被低估的宝藏
- **时间**：最近一个月最火的、经典不过时的

## 重要规则

- **依赖档案**——没有 sites/ 档案的网站跳过，提醒用户补建档，不要自行抓取
- **主题要有吸引力**——标题是用户第一眼看到的，必须让人想点
- **中英文独立写**——不是互译，各自自然
- **draft 结构严格**——必须和 picks.json 的结构一致（除了多一个 status 字段）
